# 模式：apply -- 即時申請助手

適用於候選人正在瀏覽器裡填寫申請表的場景。讀取當前頁面、載入既有職缺上下文，並針對表單中的每個問題生成客製化回答。

## 前提

- **最佳體驗：可見的 Playwright**。候選人能看到瀏覽器，Claude 也能讀取頁面內容
- **沒有 Playwright**：候選人提供截圖，或把表單問題手動貼出來

## Workflow

```
1. DETECT     -> 讀取目前 Chrome tab（截圖 / URL / 標題）
2. IDENTIFY   -> 擷取公司名 + 職缺名
3. SEARCH     -> 在既有 reports/ 中查找上下文
4. LOAD       -> 讀取完整 report + Section G（如果存在）
5. COMPARE    -> 目前職缺是否和已評估職缺一致？如果變了要提醒
6. ANALYZE    -> 辨識目前頁面可見的全部問題
7. GENERATE   -> 針對每個問題生成客製化回答
8. PRESENT    -> 用便於 copy-paste 的格式輸出
```

## 第 1 步 -- 偵測目前職缺

**有 Playwright：** 對目前頁面做 snapshot，讀取標題、URL 和可見內容。

**沒有 Playwright：** 讓候選人：

- 分享表單截圖（Read tool 可以讀圖）
- 或直接貼上表單問題
- 或至少告訴我們公司名 + 職缺名，方便查既有上下文

## 第 2 步 -- 辨識並載入上下文

1. 從頁面中擷取公司名和職缺名
2. 在 `reports/` 中依公司名搜尋（大小寫不敏感）
3. 如果找到 match，就讀取完整 report
4. 如果 report 有 Section G，也一併讀取，作為草稿基底
5. 如果沒找到，就提醒使用者並建議先跑一次快速 auto-pipeline

## 第 3 步 -- 辨識職缺是否變化

如果目前頁面職缺和既有評估不一致：

- **先提醒候選人**：「目前職缺已從 [X] 變成 [Y]。要不要重新評估，還是先沿用現有素材做適配？」
- **如果選擇適配**：在不重跑 A-F 的情況下，依新職稱重寫回答
- **如果選擇重評估**：重新執行 A-F，更新 report，重建 Section G
- **必要時更新 tracker**：如果職稱確認改了，就同步更新 `applications.md`

## 第 4 步 -- 分析表單問題

辨識目前可見的全部問題類型：

- 自由文本（例如 why this role、cover letter、why this company）
- 下拉欄位（例如 how did you hear about us、work authorization）
- Yes / No（例如 relocation、visa、是否接受 hybrid）
- 薪資欄位（期望薪資、目前薪資、總包範圍）
- 日期 / 地點欄位（最快到職日、工作地點、是否可到班 / 出差）
- 上傳欄位（履歷、求職信 PDF）

並把問題分成兩類：

- **Section G 已經答過** -> 以現有版本微調
- **全新問題** -> 依據 report + `cv.md` 生成新回答

## 第 5 步 -- 生成回答

每個問題都依以下順序生成：

1. **先用 report 上下文**：優先引用 B 區塊 proof points、F 區塊 STAR 故事
2. **再用既有 Section G**：如果已有草稿，就用它做基底並精修
3. **保持「我也在選你們」的語氣**：不是乞求，不是模板腔
4. **必須具體**：引用當前 JD 裡的職責、產品、團隊或技術棧
5. **必要時帶入 portfolio / demo**：如果表單有 “Additional info” 之類欄位，就把最相關的 proof point 放進去

**對台灣常見表單欄位的處理：**

- **薪資期望**：優先報區間，並說明更在意總包結構與職責範圍
- **最快到職日**：依真實離職交接期回答，不要任意縮短
- **工作地點 / 到班要求**：明確寫能否接受某城市、混合辦公或出差
- **身份 / work authorization**：依真實狀態填寫
- **是否接受混合辦公**：依 `profile.yml` 的 location policy 回答

**輸出格式：**

```
## [公司] -- [職缺] 的表單回答

基於：Report #NNN | Score: X.X/5 | Archetype: [類型]

---

### 1. [表單原問題]
> [可直接複製的回答]

### 2. [下一題]
> [回答]

...

---

備註：
- [關於職缺變化、資訊缺失等觀察]
- [候選人最好手動再檢查一遍的地方]
```

## 第 6 步 -- 投遞後（可選）

如果候選人確認已經送出申請：

1. 把 `applications.md` 狀態從 `Evaluated` 更新為 `Applied`
2. 把最終版回答寫回 report 的 Section G
3. 建議下一步執行 `/career-ops contact` 做 LinkedIn / Cake / 人脈外聯

## 捲動處理

如果表單問題超過一個畫面：

- 讓候選人繼續往下捲再補截圖
- 或把剩餘問題直接貼出來
- 分輪處理，直到整張表單都覆蓋完整
